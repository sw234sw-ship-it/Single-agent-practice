# Guide.md

## Step 0. 문서 기반 실행 원칙
이 프로젝트는 대화 컨텍스트가 길어져도 품질을 유지하기 위해, 아래 3문서를 "단일 진실(Single Source of Truth)"로 사용한다.
- `Plan.md`
- `ContextLog.md`
- `Checklist.md`

모든 작업 시작/재개 시 위 3문서를 먼저 읽고 진행한다.

## Step 1. 사용자 의도 이해부터 시작
사용자 요청을 받으면 구현 전에 반드시 의도 분석을 먼저 수행한다.

예시 요청: "전기화학의 기초를 다루는 웹사이트 제작"
- 목적은 교육인가, 소개인가, 홍보인가?
- 사용자 수준은 입문자인가, 전공자인가?
- 전기화학의 기초 중 무엇을 우선 전달해야 하는가?

### Intent 3-Why (에이전트 내부 질문)
1. 왜 이 사용자는 이 웹사이트를 필요로 하는가?
2. 왜 이 컨텐츠가 지금 필요한가?
3. 왜 이 형태(UI/구성)여야 목표 달성이 쉬운가?

## Step 2. 의도 파악 후 기획 준비
의도 분석이 끝나면, 목표 달성을 위한 기획으로 전환한다.

예시 기획 질문:
- 이 기초 개념을 어떻게 하면 가장 쉽게 이해시킬 수 있는가?
- 어떤 정보 구조(순서/섹션)가 학습 효율을 높이는가?
- 어떤 시각 자료(다이어그램, 비교표, 인터랙션)가 필요한가?

### Planning 3-Why (에이전트 내부 질문)
1. 왜 이 정보 구조가 사용자 목표에 가장 적합한가?
2. 왜 이 시각 자료가 학습/이해에 실질적으로 도움이 되는가?
3. 왜 이 인터랙션/화면 흐름이 단순성과 완성도를 함께 만족하는가?

## Step 3. 기획 결과물 정의
기획 단계에서 최소 아래 항목을 정리한 뒤 구현에 들어간다.
- 대상 사용자
- 핵심 목표(사용자가 얻어야 할 결과)
- 핵심 컨텐츠 목록
- 섹션 구조 및 화면 흐름
- 필요한 시각 자료/인터랙션 목록

## Step 3.5 승인 게이트 (필수)
기획 결과물을 만든 뒤, 구현 전에 반드시 사용자 승인을 받는다.
승인 전에는 코드/파일 변경을 시작하지 않는다.

승인 요청 시 반드시 포함:
- 목표 요약(1~3줄)
- 범위(in/out)
- 화면 구조(섹션/흐름)
- 작업 순서(최대 7개 단계)
- 완료 기준(수용 기준)

## Step 4. 구현 연결
기획 완료 후 `skills.md`를 따라 UI를 제작하고, 미니멀/심플 원칙과 목표 적합성을 끝까지 유지한다.

## Step 4.5 작업 중 변경 관리 (필수)
요구사항 추가/변경이 들어오면:
1) `Plan.md`에 변경 사항을 먼저 반영(변경안)
2) 사용자 승인
3) 승인 후 구현
4) `ContextLog.md`에 변경 이유/근거 기록
5) `Checklist.md` 업데이트

## Step 5. Screen Structure Definition (필수)

구현 전에 반드시 아래 구조를 정의한다.

1. Hero Section
   - 메인 메시지 1문장
   - 보조 설명 1~2줄
   - Primary CTA 1개

2. Core Section
   - 핵심 컨텐츠를 2~4 블록으로 나누어 표현
   - 각 블록은 제목 + 설명 + 시각 요소

3. Flow Section
   - 사용자가 무엇을 하게 될지 단계적 흐름 제시
   - Step 기반 구성 가능

4. Action Section
   - 사용자 행동을 유도하는 마무리 구간
   - 버튼은 1~2개 이내

이 구조 없이 구현을 시작하는 것을 금지한다.

## Execution Rule Reminder

이 문서의 기획 단계가 끝난 뒤에도,
반드시 Agent.md의 "Work Activation Protocol"을 따른다.

기획 → 승인 → 문서 생성 → 구현
순서를 절대 변경하지 않는다.

## Post-Implementation Discipline

구현 완료 후에는 반드시 Agent.md의
- Checklist Execution Engine
- Change Logging Protocol
- Quality Inspection Loop
를 수행한다.
